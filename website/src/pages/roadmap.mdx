<section id="release-plan" className={styles.sectionHeader}>
  <span className={styles.eyebrow}>Release plan</span>
  <h2>Strategic milestones</h2>
</section>

<div className={styles.releaseGrid}>
  <article id="v042" className={styles.releaseCard}>
    <div className={styles.cardHeader}>
      <span className={styles.versionBadge}>v0.4.2</span>
      <span className={styles.statusBadge}>Next</span>
    </div>
    <h3>A better first-run experience</h3>
    <p>
      The next release focuses on making setup feel natural: choose where you want to monitor pollen first, then let the integration check whether SILAM data is available for that location.
    </p>
    <ul>
      <li>Start with a Home Assistant zone or a custom point on the map.</li>
      <li>Suggest SILAM Pollen automatically for the home zone when it makes sense.</li>
      <li>Enrich the selected location with altitude and a readable place name when possible.</li>
      <li>Warn the user if the altitude looks missing or suspicious.</li>
      <li>Clearly document which external endpoints are required and which are optional.</li>
    </ul>
  </article>

  <article id="v043" className={styles.releaseCard}>
    <div className={styles.cardHeader}>
      <span className={styles.versionBadge}>v0.4.3</span>
      <span className={styles.statusBadgeMuted}>Foundation</span>
    </div>
    <h3>A cleaner technical foundation</h3>
    <p>
      This release is about long-term project health. The integration already does a lot; the next architectural step is to make the internal structure easier to evolve.
    </p>
    <ul>
      <li>Separate data access, dataset selection, diagnostics, and service status responsibilities.</li>
      <li>Keep user-facing behavior stable while improving maintainability.</li>
      <li>Prepare the codebase for richer forecast domains such as weather, dust, maps, and AI summaries.</li>
    </ul>
  </article>

  <article id="v044" className={styles.releaseCard}>
    <div className={styles.cardHeader}>
      <span className={styles.versionBadge}>v0.4.4</span>
      <span className={styles.statusBadgeMuted}>Timeline</span>
    </div>
    <h3>Forecast context from the recent past</h3>
    <p>
      Forecasts are easier to understand when users can see the recent trend. This milestone adds a short retrospective window for the custom forecast card.
    </p>
    <div className={styles.timelineSample}>
      <span>-12h</span><span>-9h</span><span>-6h</span><span>-3h</span><strong>now</strong><span>+3h</span><span>+6h</span><span>+9h</span><span>+12h</span>
    </div>
    <ul>
      <li>Show how pollen risk moved before the current moment.</li>
      <li>Keep the history window compact and dashboard-friendly.</li>
      <li>Support richer visual timelines in Absolute Forecast Card.</li>
    </ul>
  </article>

  <article id="v045" className={styles.releaseCard}>
    <div className={styles.cardHeader}>
      <span className={styles.versionBadge}>v0.4.5</span>
      <span className={styles.statusBadgeMuted}>Weather</span>
    </div>
    <h3>More weather context around pollen</h3>
    <p>
      Pollen is strongly shaped by weather. This release expands the forecast experience with weather-related context where the available data allows it.
    </p>
    <ul>
      <li>Bring temperature, wind, humidity, precipitation, and cloud context closer to the pollen forecast.</li>
      <li>Use fields already available in the current forecast where practical.</li>
      <li>Leave room for a separate weather dataset later if it becomes the better path.</li>
      <li>Make the custom card more useful for real daily decisions.</li>
    </ul>
  </article>

  <article id="v046" className={styles.releaseCard}>
    <div className={styles.cardHeader}>
      <span className={styles.versionBadge}>v0.4.6</span>
      <span className={styles.statusBadgeMuted}>Data access</span>
    </div>
    <h3>Clean summaries for automations and dashboards</h3>
    <p>
      This milestone creates a stable, compact way to ask the integration what matters now, what is coming next, and whether the data source is healthy.
    </p>
    <ul>
      <li>Expose clean current, forecast, and service-status summaries.</li>
      <li>Make automation and notification use cases easier.</li>
      <li>Keep raw internal data private and avoid exposing oversized payloads.</li>
    </ul>
  </article>

  <article id="v047" className={styles.releaseCard}>
    <div className={styles.cardHeader}>
      <span className={styles.versionBadge}>v0.4.7</span>
      <span className={styles.statusBadgeMuted}>Assist</span>
    </div>
    <h3>AI-ready pollen answers</h3>
    <p>
      Home Assistant Assist and LLMs should be able to answer pollen questions in plain language without reading raw internal attributes.
    </p>
    <ul>
      <li>Provide safe, compact context for questions like “How bad is pollen today?”</li>
      <li>Explain risk, primary allergens, best and worst periods, data freshness, and service health.</li>
      <li>Avoid exposing precise coordinates, full data URLs, debug internals, or heavy raw forecast payloads by default.</li>
    </ul>
  </article>
</div>
